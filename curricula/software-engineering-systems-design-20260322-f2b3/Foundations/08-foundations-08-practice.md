## Exercises

**Exercise 1**
A popular video-sharing platform displays a "view count" on each video. When a user watches a video, the system increments the count. Is it acceptable for a user in Japan to see a view count of 5,012 for a few minutes, while a user in Brazil simultaneously sees 5,010 for the same video? Which consistency model is more appropriate for this feature, and why?

**Exercise 2**
A user is changing their password on a financial services website. After successfully submitting the new password, they are redirected to the login page. They immediately try to log in with their new password. Which consistency model is required for the authentication system in this scenario? Explain the consequence of choosing the wrong model.

**Exercise 3**
You are designing a notification system for an e-commerce app. The system sends two types of notifications: "Order Shipped" and "Recommended for You." An "Order Shipped" notification is triggered by an internal logistics system update. A "Recommended for You" notification is generated by a background analytics job. Compare the consistency requirements for delivering these two types of notifications. Which one can tolerate more delay and potential inconsistency, and why?

**Exercise 4**
A global cloud storage service (like Dropbox or Google Drive) allows users to upload files via a desktop client. When a user adds a file to their synchronized folder on their laptop, the file is uploaded to the cloud and then downloaded to their other devices (e.g., their phone and desktop computer). Often, the file appears on the other devices seconds or even minutes later. Why is eventual consistency a reasonable choice for a file synchronization feature, and what primary technical trade-off is being made to achieve this?

**Exercise 5**
A company is designing a distributed database for a hotel reservation system. To improve performance (reduce latency) for their global customer base, they replicate the data across servers in North America, Europe, and Asia. A network failure partitions the Asian datacenter from the others. During this partition, a customer in Europe books the last available room in a Paris hotel. Almost simultaneously, a customer in Asia attempts to book the same room. If the system is designed to prioritize availability, what is likely to happen? Connect this outcome to the concept of strong vs. eventual consistency.

**Exercise 6**
You are the lead engineer for a new ride-sharing application. The app has a feature that shows drivers the current "surge pricing" multiplier for different city areas, which is updated every minute based on rider demand and driver supply. The app also processes the final payment at the end of a ride. For this system to be successful, it must be highly available and scale to millions of users.

Propose a consistency model for (1) the surge pricing display and (2) the final payment processing. Justify your choices by explaining how they support the system's goals of high availability and scalability, while considering the impact on user experience and data correctness.

---

## Answer Key

**Answer 1**
The "view count" feature is a good candidate for **eventual consistency**.

**Reasoning:**
A small, temporary discrepancy in the view count between different users in different parts of the world has a negligible impact on the user experience. It is not a critical piece of data that must be perfectly synchronized at all times. Choosing eventual consistency allows the system to be more performant and available, as it doesn't need to wait for every server worldwide to agree on the exact count before displaying it. The system can batch updates and propagate them gradually, reducing system load.

**Answer 2**
The authentication system requires **strong consistency**.

**Reasoning:**
After a user changes their password, they expect to be able to use it immediately. The read operation (login attempt) must reflect the absolute latest write operation (password update). If the system were eventually consistent, the login attempt might be routed to a replica that hasn't yet received the password update. This would cause the user's new, correct password to be rejected, leading to a frustrating and confusing user experience where it seems the password change failed.

**Answer 3**
The "Order Shipped" notification has a higher consistency requirement than the "Recommended for You" notification.

**Reasoning:**
- **"Order Shipped":** This notification is transactional and directly related to a user's specific action (placing an order). A significant delay or failure to deliver this notification could cause user anxiety and increase customer support inquiries. While it doesn't need to be instantaneous, it should be highly reliable and reflect the true state of the logistics system in a timely manner.
- **"Recommended for You":** This notification is not based on a real-time user action and is not critical. If the recommendation is based on data that is a few hours old, or if the notification is delayed, the user experience is not significantly impacted. This feature can easily tolerate eventual consistency, allowing the system to prioritize efficiency and batch processing for the analytics jobs.

**Answer 4**
Eventual consistency is a reasonable choice because the core user expectation is that the file *will* sync, not that it will sync *instantaneously*.

**Reasoning:**
The primary trade-off being made is sacrificing immediate consistency for higher **availability** and better **performance/throughput**.
- **Availability:** If the system required strong consistency, a network issue preventing a user's phone from contacting the master database could prevent them from uploading a file on their laptop. With eventual consistency, the upload can succeed locally and sync later.
- **Performance:** Forcing every file write to be instantly replicated to all data centers worldwide before confirming success would introduce significant latency for the user, making the application feel slow. Eventual consistency allows the initial upload to be confirmed quickly, with replication happening in the background.

**Answer 5**
If the system prioritizes availability, it is likely that **both customers will successfully book the room**, resulting in a double-booking. This is a classic outcome of an **eventually consistent** system operating under a network partition.

**Reasoning:**
- To remain available, the servers in both the European and Asian datacenters must continue to accept reservation requests even though they cannot communicate with each other.
- Each datacenter's replica would show one room available and would independently process a booking transaction, considering it successful.
- This creates an inconsistent state between the replicas. Once the network partition is resolved, the system will need to reconcile the data and discover the conflict (the double-booking). At this point, the business will have to intervene to fix the problem (e.g., by apologizing to one customer and offering them a different room). This prioritizes uptime over the correctness guaranteed by strong consistency.

**Answer 6**
Different consistency models should be used for the two features.
1.  **Surge Pricing Display: Eventual Consistency**
2.  **Final Payment Processing: Strong Consistency**

**Reasoning:**
- **Surge Pricing Display (Eventual):** The surge multiplier is informational and is updated periodically. If a user in one part of the city sees a multiplier that is a minute out of date compared to another user, the impact is low. The data is not transactional. Choosing eventual consistency for this feature allows the system to scale massively. Surge data can be calculated and then broadcast to regional caches without waiting for global confirmation. This improves **availability** (the map and prices are always visible) and reduces **latency** for users, supporting the system's primary goals.

- **Final Payment Processing (Strong):** A user's account balance and the final charge are critical, transactional data. The system must guarantee that a payment transaction is processed correctly and completely once and only once. Using eventual consistency here could lead to catastrophic errors like double-charging a user or failing to record a payment. Strong consistency ensures that when a payment is processed, the read of the user's payment method and the write to the ledger are atomic and immediately reflected across the system, guaranteeing data correctness at the cost of slightly higher latency during the transaction itself. This trade-off is essential for a financial transaction.
