## Capstone: Mechanics — Software Engineering with AI and AI Engineering

### The Scenario
You are a new AI Engineer at **SecureNet Analytics**, a cybersecurity firm. The company's Security Operations Center (SOC) is struggling to keep up with the volume of alerts generated by their traditional, rule-based network monitoring systems. Many of these alerts are false positives, wasting valuable analyst time. To address this, the company wants to develop a machine learning-based Network Intrusion Detection System (NIDS) to more accurately classify network traffic.

A senior data scientist has already performed an initial analysis and provided you with a dataset (`network_traffic.csv`). This dataset contains records of individual network connections, each described by 40+ features (e.g., `duration`, `protocol_type`, `src_bytes`, `dst_bytes`) and labeled as either `normal` or an `attack` type. Your assignment is to build a deployable prototype of this NIDS. The end goal is a portable, reliable service that can receive data for a new network connection and predict whether it's malicious. The system must also be designed with long-term maintenance in mind, as attackers constantly change their methods.

### Your Tasks
1.  **Build a Data Preprocessing Pipeline:** Create a robust, repeatable pipeline that takes the raw `network_traffic.csv` data and prepares it for model training. This pipeline must handle categorical features, scale numerical features appropriately, and split the data into training and testing sets. Document the key transformation steps and the rationale behind them.

2.  **Train and Evaluate a Classifier:** Using a validation strategy such as k-fold cross-validation, train a classification model (e.g., Logistic Regression, Random Forest, or Gradient Boosting). Systematically tune at least two key hyperparameters to improve performance. Justify your choice of the primary evaluation metric (e.g., accuracy, precision, recall, F1-score) based on the project's real-world consequences. Save the final, trained model artifact.

3.  **Develop a Prediction Service API:** Create a simple web service with a RESTful API. This service must load your saved model artifact and expose a single endpoint (e.g., `/predict`). This endpoint should accept a JSON object containing the features for a single network connection and return a prediction (e.g., `{"prediction": "attack"}`) and a confidence score.

4.  **Containerize the Application:** Write a `Dockerfile` to containerize your entire prediction service. The container image must include all necessary dependencies, your trained model artifact, and the code for the API server. The final container should be self-contained and runnable on any machine with Docker installed.

5.  **Design a Model Monitoring Strategy:** The security landscape is constantly changing, meaning the patterns in network traffic will evolve (a phenomenon known as "concept drift"). Propose a technical strategy for monitoring the deployed model in a production environment. Describe what specific data you would collect and which metric(s) you would track to detect a significant drop in performance or a shift in the input data distribution.

### What Good Work Looks Like
*   **Systematic and Justified Decisions:** Your choices are not arbitrary. You clearly explain *why* you chose a specific scaling method, evaluation metric, hyperparameter tuning range, or drift detection technique, connecting your reasoning back to the goals of building a reliable security tool.
*   **Focus on Reproducibility:** Your work emphasizes automation and repeatability. The data pipeline is a script, not a series of manual steps. The model training process is documented, and the entire application is encapsulated in a container, demonstrating an understanding of production-ready engineering principles.
*   **End-to-End Thinking:** You demonstrate a clear understanding of the entire model lifecycle, not just the training phase. Your solution connects the dots from raw data to a deployable, containerized API and includes a forward-looking plan for post-deployment monitoring.
*   **Pragmatic Problem-Solving:** Your solution is practical and focused on the core problem. You build a simple, functional API and a straightforward Dockerfile, recognizing that the goal is a working prototype, not an over-engineered system.
